## Arena AI Agents

**FEAT-FP-WCA-001: Wave Client Agent — Prompt, Commands, and Tool-Orchestration Refactor** *(updated: 2026-05-02 by implement)*
Source spec: [feature_wave_client_agent_refactor.md](.spec-lite/features/feature_wave_client_agent_refactor.md)
The Wave Client agent now uses a canonical inline prompt contract with mandatory operational response sections (`TL;DR`, `What I checked`, `Findings`, `Recommended action`, `Next Steps`) and depth tiers (`Quick`, `Default`, `Deep`). Workspace answers are explicitly tool-evidence-driven: the agent must not fabricate workspace entities and must state that it cannot verify without MCP/tool output when data is unavailable. The Wave command surface is now synchronized across parser and UI metadata (`/help`, `/collections`, `/requests`, `/environments`, `/flows`, `/tests`, `/run-flow`, `/run-tests`) with deterministic slash-command normalization, free-form intent hinting, and confirmation-first run behavior (`list_* -> confirm -> run_*`) for flow/test execution.

**Web Expert Agent — Inline System Prompt** *(updated: 2026-04-30 by implement)*
Source spec: [feature_web_expert_inline_prompt.md](.spec-lite/features/feature_web_expert_inline_prompt.md)
The Web Expert Agent now uses its full 245-line system prompt at runtime. The prompt is inlined as the `WEB_EXPERT_SYSTEM_PROMPT` constant in `packages/arena/src/agents/webExpertAgent.ts` — the canonical single source of truth (no companion `.md` file). The prompt covers all expertise domains (Network & Transport, HTTP, Real-Time, RPC, API Design, Security, Web Platform), a four-tier reference source hierarchy (IETF RFCs → MDN → ecosystem docs → Hacker News exception), citation rules, GFM response formatting rules, and all command definitions. The `systemPrompt` config option overrides the inlined constant per-instance.

## WebSocket & SSE

**FEAT-010: Collection Persistence for WS/SSE** *(updated: 2026-04-15 by implement)*
WebSocket and SSE requests are now persisted in collection JSON files with full protocol fidelity. The save pipeline from `RequestEditor` through app handlers and adapters accepts `AnyCollectionRequest` (the discriminated union of HTTP, WS, and SSE) instead of HTTP-only `CollectionRequest`. `CollectionService` applies protocol-aware normalization on load (`normalizeRequestOnLoad`) and sanitization on save (`sanitizeRequestForSave`): WS payloads strip HTTP-only fields (`method`, `body`, `validation`); SSE payloads strip `validation` and default `method` to `GET`; legacy entries without a `protocol` field are normalized to HTTP. Runtime-only `sourceRef` metadata is never persisted. `extractRequestFromItem` and `requestToCollectionItem` in `collectionParser.ts` now use generic spreads to preserve all protocol-specific fields during round-trips. Mixed-protocol collections (HTTP + WS + SSE + legacy) load and save correctly with stable IDs. Covered by 15 new core tests (collectionParser round-trip) and 20 new shared tests (normalization, sanitization, persistence round-trip).

**FEAT-009: Web Adapter + Server Routes** *(updated: 2026-04-15 by implement)*
The web platform now has a full `IRealtimeAdapter` implementation via `createWebRealtimeAdapter` in `packages/web/src/adapters/webRealtimeAdapter.ts`, with synchronous handle creation (`connectWebSocket`, `connectSse`) and server-backed lifecycle operations (`/api/ws/connect`, `/api/ws/disconnect`, `/api/ws/send`, `/api/sse/connect`, `/api/sse/disconnect`). The server now exposes dedicated realtime route modules (`packages/server/src/routes/realtime-ws.ts`, `packages/server/src/routes/realtime-sse.ts`) that delegate connection management to shared services (`webSocketService`, `sseService`) and relay runtime updates through the existing push channel as `ws.*` and `sse.*` event types. `webAdapter.ts` now wires these push events into active handle registries (`wsHandles`, `sseHandles`) and exposes `adapter.realtime` so WS/SSE UI hooks work end-to-end in the web app. FEAT-009 includes comprehensive route tests (17 server tests) and web adapter tests (47 FEAT-009-focused tests across `webRealtimeAdapter.test.ts` and `webAdapter.test.ts`).

**FEAT-001: Protocol Types & Core Data Model** *(updated: 2026-04-14 by implement)*
Three request protocols are now defined as first-class types in Wave Client: `CollectionRequest` (HTTP, `protocol?: 'http'`), `WsCollectionRequest` (`protocol: 'ws'`), and `SseCollectionRequest` (`protocol: 'sse'`). These form the `AnyCollectionRequest` discriminated union used by `CollectionItem.request`. Type guard functions (`isHttpRequest`, `isWsRequest`, `isSseRequest`, `getRequestProtocol`) live in `packages/core/src/utils/requestTypeGuards.ts` and are exported from the core package index. All pre-existing collections without a `protocol` field are treated as HTTP (backward compatible — no migration needed). 35 unit tests in `packages/core/src/test/types/requestTypeGuards.test.ts` cover all guards, serialization round-trips, and backward compatibility.

**FEAT-002: Realtime State Slice** *(updated: 2026-04-14 by implement)*
Per-tab, in-memory realtime state for WebSocket and SSE connections is now part of the core Zustand store. New types in `packages/core/src/types/realtime.ts` define `ConnectionStatus`, `WsMessage`, `SseEvent`, `RealtimeTabState`, and `RealtimeStateByTabId`, all exported from `@wave-client/core`. A new Zustand slice (`createRealtimeSlice`) composed into `useAppStateStore` provides: idle state management (`ensureRealtimeTabState`, `getRealtimeState`, `setRealtimeStatus`, `setRealtimeConnectionId`, `setRealtimeResponseHeaders`, `setRealtimeError`, `resetRealtimeTabState`, `removeRealtimeTabState`); timeline actions (`appendWsMessage`, `appendSseEvent`, `clearRealtimeTimeline`, `setSseEventFilter`, `getFilteredSseEvents`); and automatic lifecycle wiring in `createRequestTabsSlice` so that loading, protocol-switching, clearing, and closing tabs always keep `RealtimeStateByTabId` consistent — WS/SSE tabs get a fresh idle entry on load, HTTP tabs never have one, protocol switches discard stale state. `TabData.protocol` is now typed as `RequestProtocol` (not a free-form string), with protocol-aware section defaults and request serialization (`getCollectionRequest()` returns the correct discriminated-union member). All realtime state is ephemeral (never persisted). 51 new unit tests cover TASK-001 through TASK-004.

**FEAT-005: SSE Service** *(updated: 2026-04-15 by implement)*
`SseService` in `packages/shared/src/services/SseService.ts` manages the full lifecycle of Server-Sent Events connections from the Node.js runtime (VS Code extension host or server). It opens SSE streams via `axios` with `responseType: 'stream'` (not the browser `EventSource` API — required for POST-based SSE, custom headers, and auth); parses the SSE wire protocol incrementally using an internal `SseFrameParser` class (handles `event:`, `data:`, `id:`, `retry:`, comments, partial chunks, `\r\n` endings, and all edge cases from the HTML Living Standard spec); routes parsed events and lifecycle transitions to event-driven `SseConnectionHandle` callbacks (`onEvent`, `onStatusChange`, `onError`, `onHeaders`); and closes connections via `disconnect()` using `AbortController`. Auth support (API key, Bearer, Basic — Digest is skipped with a warning), TLS/proxy via `storeService.getHttpsAgentForUrl()`, and default SSE headers (`Accept: text/event-stream`, `Cache-Control: no-cache`) are applied on every request. User-supplied headers override defaults. POST-based SSE with request body is supported. A maximum of 10 concurrent connections is enforced. Two new types were added to `@wave-client/core` and exported: `SseConnectionConfig` and `SseConnectionHandle`. The injectable `setSseAuthServiceFactory()` enables isolated unit testing. 44 unit tests in `packages/shared/src/test/services/SseService.test.ts` cover the frame parser (T01–T15), connection lifecycle (T16–T37), and auth/proxy integration (T38–T44).

**FEAT-004: WebSocket Service** *(updated: 2026-04-20 by fix)*
`WebSocketService` in `packages/shared/src/services/` manages the full lifecycle of WebSocket connections from the Node.js runtime (VS Code extension host or server). It opens connections with custom headers, query params, auth (API key, Bearer, Basic — Digest is skipped with a warning), and TLS/proxy settings via `storeService.getHttpsAgentForUrl()`; routes incoming messages and status transitions to event-driven `WsConnectionHandle` callbacks (`onMessage`, `onStatusChange`, `onError`, `onHeaders`); lets callers send text messages via `sendMessage()` (sent messages are emitted as `direction: 'sent'`); and closes connections via `disconnect()`. A maximum of 10 concurrent connections is enforced. Three new types were added to `@wave-client/core`: `Unsubscribe` (`() => void`), `WsConnectionConfig` (connection parameters), and `WsConnectionHandle` (event-driven handle). The injectable `setWsAuthServiceFactory()` enables isolated unit testing. A race condition fix now caches handshake response headers and immediately replays them to late `onHeaders` subscribers, so the Response Headers pane reliably populates even when the socket opens before UI listeners are attached. `useWsConnection.sendMessage()` now surfaces explicit user-facing errors for unavailable adapter/disconnected state, falls back to the tab's stored `connectionId` when the local hook handle is unavailable (for example after component remount), and appends a local sent message on successful send so the Messages pane updates immediately. To avoid duplicate sent entries, WS push relays now ignore `direction: 'sent'` echoes in both server and VS Code bridge paths. 28 unit tests in `packages/shared/src/test/services/WebSocketService.test.ts`, 19 hook tests in `packages/core/src/test/hooks/useWsConnection.test.ts`, 11 server route tests in `packages/server/src/test/routes/realtime-ws.test.ts`, and 24 VS Code handler tests in `packages/vscode/src/test/handlers/MessageHandlerRealtime.test.ts` cover public methods, auth resolution, connection limits, race handling, and send-path robustness.

**FEAT-007: SSE UI — Connection & Event Stream** *(updated: 2026-04-15 by implement)*
`SseEventTimeline` in `packages/core/src/components/common/SseEventTimeline.tsx` renders a scrollable, timestamped event log for SSE connections. Each event row shows a purple badge with the event name, a `<pre>` block for the event data (truncated to 2 000 chars with a warning if exceeded), and an `HH:MM:SS` timestamp. When the event list is empty, a `RadioIcon` empty state is shown with "No events yet" and a "Connect to an SSE endpoint to see events here." subtext. A Radix `Select` filter dropdown (populated from unique event names seen so far) and a `Trash2Icon` Clear button appear in the toolbar once at least one event exists. Filtering state is persisted to the Zustand store via `setSseEventFilter`; derived filtered events are computed in the component (not inside the Zustand selector) to avoid new array references on every render. Auto-scroll follows new events via a `bottomRef`. `useSseConnection` in `packages/core/src/hooks/useSseConnection.ts` mirrors `useWsConnection` for the SSE lifecycle: it reads the active tab's SSE request from the store, calls `adapter.realtime.connectSse()` with a `SseConnectionConfig` (URL, method, body, headers, query params), and wires the handle's `onEvent`, `onStatusChange`, `onError`, and `onHeaders` callbacks to the Zustand store actions. `SseOutputArea` now renders `SseEventTimeline` instead of the old placeholder stub. Both `RequestEditor` components (VS Code and web) now import `useSseConnection` and route `ConnectionControls.onConnect`/`onDisconnect` to the SSE hook when `protocol === 'sse'`. 25 new unit tests cover `SseEventTimeline` (13) and `useSseConnection` (12).

**FEAT-008: VS Code Adapter — Realtime Integration** *(updated: 2026-04-15 by implement)*
The VS Code platform adapter now fully implements `IRealtimeAdapter` so the WS/SSE UI (FEAT-006, FEAT-007) works end-to-end inside VS Code. Two layers were added: (1) **Extension host** — `MessageHandler.ts` handles `ws.connect`, `ws.disconnect`, `ws.send`, `sse.connect`, and `sse.disconnect` postMessages from the webview; each handler delegates to `webSocketService` / `sseService` (from `@wave-client/shared`), awaits the async `connect()` calls, and pushes back per-connection lifecycle events (`ws.message`, `ws.status`, `ws.headers`, `ws.error`, `sse.event`, `sse.status`, `sse.headers`, `sse.error`) without a `requestId`. Null returns from `connect()` (pre-flight validation failures) are converted to error + status push events. A `public dispose()` method disconnects all active handles on panel close. `extension.ts` wires `panel.onDidDispose` to call `messageHandler.dispose()`. (2) **Webview side** — `vsCodeAdapter.ts` gains `createWsVSCodeHandle` and `createSseVSCodeHandle` factory functions that create internal handle types extending `WsConnectionHandle`/`SseConnectionHandle` with `dispatch*` methods for push event routing. `createVSCodeRealtimeAdapter` implements `IRealtimeAdapter`: `connectWebSocket`/`connectSse` return handles synchronously (creating them eagerly and posting `ws.connect`/`sse.connect` to the host), while `disconnectWebSocket`, `sendWebSocketMessage`, `disconnectSse` use `sendAndWait` for request/response. The `handleMessage` switch routes the 8 push-event types to the correct handle via the `wsHandles`/`sseHandles` maps. `adapter.realtime` is now populated on the returned adapter object; the `cleanup()` function fire-and-forget disconnects all active handles. 52 new unit tests in `packages/vscode/src/test/handlers/MessageHandlerRealtime.test.ts` (23 tests) and `packages/vscode/src/test/adapters/vsCodeRealtimeAdapter.test.ts` (29 tests) cover all handler paths, push event dispatch, unsubscribe, stale event handling, concurrent handles, timeouts, and disposal.

**FEAT-003: Protocol Selector & Conditional Editor Shell** *(updated: 2026-04-15 by implement)*
The request editor is now protocol-aware. A new `ProtocolSelector` dropdown (Radix Select with Globe/Cable/Radio icons) appears in the toolbar of both `packages/vscode/src/webview/RequestEditor.tsx` and `packages/web/src/components/RequestEditor.tsx`, letting users switch any tab between HTTP, WebSocket, and SSE. The editor shell morphs on selection: the HTTP method dropdown and Send button are hidden for WS/SSE; a `ConnectionControls` component (status dot + label + Connect/Disconnect `PrimaryButton`) replaces them; request section tabs narrow to `[Params, Headers]` for WS and `[Params, Headers, Body]` for SSE; and the response output area renders `WsOutputArea` (Messages + Response Headers) or `SseOutputArea` (Events + Response Headers) instead of the HTTP response viewer. `TabsBar` now shows a teal `WS` or purple `SSE` badge instead of the HTTP method abbreviation for non-HTTP tabs. `ResponseSectionTab` is extended with `'Messages'` and `'Events'`; `getDefaultResponseSection`, `getRequestTabsForProtocol`, and `getResponseTabsForProtocol` helpers are exported from `@wave-client/core`. All four new components are platform-agnostic and live in `packages/core`. HTTP behavior is 100% unchanged. 49 new unit tests cover all new components and type helpers (TASK-001 through TASK-006).

## Validation

**FEAT-001: Validation Engine Enhancement** *(updated: 2026-04-05 by implement)*
The validation engine in `packages/shared/src/utils/validationEngine.ts` now uses `jsonpath-plus` for full JSONPath expression evaluation (recursive descent `$..`, wildcards `[*]`, array slicing, filter expressions) and `ajv` for real JSON Schema validation (draft-07 compatible). The old naive dot-notation splitter and parse-only stub have been replaced. A new `validateJsonSchemaString(schemaString)` helper is exported from `@wave-client/shared` — it returns `{ valid: boolean; errors?: string[] }` and is intended for UI-side pre-validation of user-provided schema strings before saving a rule. All JSONPath evaluation uses `eval: false` to block script injection. The `ajv` instance is a module-level singleton (`allErrors: true`) for performance. All existing tests continue to pass; 28 new test cases cover jsonpath-plus features, ajv schema validation, and `validateJsonSchemaString` edge cases (237 tests total).

**FEAT-003: Validation UX Polish — Tooltips, Placeholders & Schema Validation** *(updated: 2026-04-05 by implement)*
`ValidationRuleEditor` (the unified rule-editing form from FEAT-002) now has three layers of UX polish: (1) **Field-level tooltips** — a `FieldTooltip` internal helper component wraps a Radix UI `Tooltip` around an `InfoIcon`; body rules show a tooltip on the category selector explaining the content-type/JSONPath/operator relationship; `BodyFields` also shows tooltips on the JSON Path label ("Use JSONPath expressions…") and the JSON Schema label ("Enter a valid JSON Schema draft-07 object…"). (2) **Operator-aware placeholders** — in `HeaderFields`, the `Expected Value` input shows a regex-specific example placeholder (`e.g., ^[0-9a-f]{8}-...`) when the operator is `matches_regex`, and a generic example otherwise. (3) **Real-time JSON Schema validation indicator** — in `BodyFields` when the operator is `json_schema_matches`, the schema textarea is followed by a live indicator: a green `CheckIcon` + "Valid JSON Schema" when the schema compiles, or a red `XIcon` + the first ajv error message when it does not. The indicator updates on every keystroke via `useEffect`. A local `validateJsonSchemaString` utility was added to `packages/core/src/utils/schemaValidation.ts` (using `ajv ^8.17.1` added to `packages/core` deps) rather than importing from `@wave-client/shared`, to avoid a circular dependency (`shared → core → shared`). 10 new component tests (T1–T10) and 9 utility tests cover all new behaviour.

OpenAPI/Swagger import now uses `@scalar/openapi-parser` for robust parsing and inline `$ref` dereferencing instead of hand-rolled schema types. ## Collections UI

**FEAT-002: Collections Pane — Menu Icons, Working Delete with Confirmation, Rename** *(updated: 2026-04-09 by implement)*
Collection, folder, and request rows in the Collections pane now all expose consistent iconized action menus (Run / Rename / Delete). Inline rename is supported at every level with sibling-level uniqueness checks and Enter/Escape shortcuts. Delete actions are guarded by a shared `ConfirmDialog`. All mutations are persisted through the adapter boundary (`saveCollection` / `deleteCollection` / `deleteRequestFromCollection`) and reflected in Zustand state via `updateCollection` / `removeCollection`. Adapter parity gaps closed: `deleteRequestFromCollection` in `vsCodeAdapter` now uses `sendAndWait` with `responseDataMap` correlation; `MessageHandler` handles the new `deleteCollection` and `deleteRequestFromCollection` message types; `webAdapter` calls the new `DELETE /api/collections/:filename/items/:itemId` server route; `CollectionService.deleteItem` and the server route were added in `packages/shared` and `packages/server`. New tree utilities (`renameItemInTree`, `removeItemFromTree`, `getSiblingsAtPath`) were added to `collectionParser.ts`. Covered by 22 new component tests across `CollectionsPane.test.tsx` (9 tests) and `CollectionTreeItem.test.tsx` (13 tests).

**FEAT-005: Environments Pane — Add Context Menu with Rename and Delete** *(updated: 2026-04-09 by implement)*
Each environment row now exposes a hover-visible three-dots (`MoreVertical`) dropdown menu with Rename and Delete actions. Clicking Rename opens an inline `<Input>` seeded with the environment's current name; pressing Enter or blurring commits, Escape cancels. On commit, the handler trims the input, skips the adapter call when the name is unchanged, and enforces case-insensitive uniqueness against all other environments — duplicate names surface an error notification and abort without touching the adapter. On success, `storageAdapter.saveEnvironment` is called first, and the Zustand store is updated via `updateEnvironment` only after the adapter confirms. Adapter failures surface via `notification.showNotification('error', ...)` without mutating the store. Delete is confirm-gated via the shared `useConfirmDialog` hook: `storageAdapter.deleteEnvironment` is called only after explicit user confirmation; the store is mutated via `removeEnvironment` only on adapter success; errors surface without removing the row. `<ConfirmDialogComponent />` is rendered in all four branches (loading, error, empty, populated). Covered by 14 component tests in `EnvironmentsPane.test.tsx`.

**FEAT-004: Flows Pane — Add Delete Confirmation** *(updated: 2026-04-09 by implement)*
Flow deletion in the Flows pane is now guarded by the shared `useConfirmDialog` hook: clicking Delete opens a titled confirmation dialog; the adapter call (`deleteFlow`) only proceeds on explicit user confirmation; the Zustand store is mutated (`removeFlow`) only on adapter success; adapter errors surface via `notification.showNotification('error', ...)`. All existing row behaviors are preserved: the hover Run button remains present for flows with nodes, inline rename with uniqueness validation continues to work, and the `Running...` label hides the action controls for flows mid-run. Covered by 15 component tests in `FlowsPane.test.tsx`.

**FEAT-003: Test Lab Pane — Remove Run Hover Button, Add Delete Confirmation** *(updated: 2026-04-09 by implement)*
Test suite rows in the Test Lab pane no longer render a per-row hover run shortcut button (PlayIcon). The `onTestSuiteRun` prop has been removed from `TestLabPane` and its callers (`ConfigPanel` in both vscode and web packages). Delete is now guarded by the shared `useConfirmDialog` hook: clicking Delete opens a titled confirmation dialog; the adapter call (`deleteTestSuite`) only proceeds on explicit user confirmation; the Zustand store is mutated (`removeTestSuite`) only on adapter success; adapter errors surface via `notification.showNotification('error', ...)`. The `Running...` status label and inline rename behavior are preserved. Covered by 13 new component tests in `TestLabPane.test.tsx`.

## Run Reporting

**HTML Run Report Interactivity and Response Rendering** *(updated: 2026-05-08 by fix)*
Source spec: [feature_run_reporting_foundation.md](.spec-lite/features/feature_run_reporting_foundation.md)
HTML reports include client-side filtering and bulk controls: the entire Total / Passed / Failed / Skipped summary tile is the click target (not just the label), clicking the same status again clears the filter, and Expand All / Collapse All buttons toggle every request detail card in one click. The summary tile counts are derived from the rendered items at build time using the same classification the per-card filter uses, so a request with HTTP 200 but failed validation is counted as Failed in the tile (matching what the Failed filter shows). A search field above the tiles filters top-level cards by name, method, URL, and folder. Request cards display the full name/URL via native `title` tooltips when truncated, the expanded body uses a slightly recessed background to visually nest under the summary row, and the active tab is brighter and bolder than inactive tabs. Response and validation payloads render in horizontally-scrollable `<pre><code>` blocks (long single-line JSON no longer wraps into an unreadable wall of text). Status indicator chips show colored text only — the icon glyphs that previously appeared inside the chips have been removed. Response rendering decodes base64-encoded text responses before pretty-printing/escaping, and card headers surface validation status for successful request runs.
