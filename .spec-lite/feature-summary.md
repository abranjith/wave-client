## Arena AI Agents

**FEAT-FP-WCA-001: Wave Client Agent — Prompt, Commands, and Tool-Orchestration Refactor** *(updated: 2026-05-02 by implement)*
Source spec: [feature_wave_client_agent_refactor.md](.spec-lite/features/feature_wave_client_agent_refactor.md)
The Wave Client agent now uses a canonical inline prompt contract with mandatory operational response sections (`TL;DR`, `What I checked`, `Findings`, `Recommended action`, `Next Steps`) and depth tiers (`Quick`, `Default`, `Deep`). The Wave command surface is now synchronized across parser and UI metadata (`/help`, `/collections`, `/requests`, `/environments`, `/flows`, `/tests`, `/run-flow`, `/run-tests`) with deterministic slash-command normalization, free-form intent hinting, and confirmation-first run behavior (`list_* -> confirm -> run_*`) for flow/test execution.

**Web Expert Agent — Inline System Prompt** *(updated: 2026-04-30 by implement)*
Source spec: [feature_web_expert_inline_prompt.md](.spec-lite/features/feature_web_expert_inline_prompt.md)
The Web Expert Agent now uses its full 245-line system prompt at runtime. The prompt is inlined as the `WEB_EXPERT_SYSTEM_PROMPT` constant in `packages/arena/src/agents/webExpertAgent.ts` — the canonical single source of truth (no companion `.md` file). The prompt covers all expertise domains (Network & Transport, HTTP, Real-Time, RPC, API Design, Security, Web Platform), a four-tier reference source hierarchy (IETF RFCs → MDN → ecosystem docs → Hacker News exception), citation rules, GFM response formatting rules, and all command definitions. The `systemPrompt` config option overrides the inlined constant per-instance.

## VS Code Extension

**Activity Bar View Wiring and Icon Compliance** *(updated: 2026-06-07 by fix)*
The full Wave Client app always opens as an editor-area webview panel (`ViewColumn.One`) — the editor gives the full-fledged app the space it needs, which the narrow Activity Bar sidebar cannot. A single shared panel instance is reused, so both entry points land on the identical view: the `waveclient.open` command (and its `Ctrl+Alt+W` / `Cmd+Alt+W` keybinding) opens the panel or reveals it if already open, and the Activity Bar icon does the same. The Activity Bar `wave-client-view` is now a lightweight launcher rather than the app host: its `WebviewViewProvider` renders a small branded "Open Wave Client" view (theme-aware logo + button) and auto-opens/reveals the full editor panel whenever the sidebar becomes visible (on icon click), with the button as an explicit fallback. The interactive `MessageHandler` (HTML injection, queued webview-message replay, and disposal-on-close) is wired to the editor panel; the panel singleton is cleared on dispose so a later open recreates it. The Activity Bar container uses a dedicated monochrome SVG (`packages/vscode/assets/wave-client-activitybar.svg`) compatible with VS Code Activity Bar theming, avoiding fallback or misrendered visuals from full-color logos.

## Authentication

**FEAT-FP-AUTH-001: Auth Flows Enhancement — OAuth2 Client Credentials, Authorization Code (PKCE), HMAC** *(updated: 2026-06-13 by implement)*
Source spec: [feature_auth_flows_enhancement.md](.spec-lite/features/feature_auth_flows_enhancement.md)
The Auth Store now supports three new industry-standard auth types in addition to the existing API Key, Basic, Digest, and OAuth2 Refresh types. **OAuth2 Client Credentials** (`oauth2ClientCredentials`) fetches machine-to-machine tokens at request time with optional scope/audience parameters, caches by TTL when `expires_in` is provided, and skips caching otherwise. **OAuth2 Authorization Code with PKCE** (`oauth2AuthorizationCode`) has a two-phase model: the wizard drives an interactive PKCE acquisition flow (generate code-verifier + S256 challenge, open authorization URL in browser, paste callback URL or bare code, exchange for tokens); the service applies the stored access token and refreshes transparently via the refresh token when expired. **HMAC** (`hmac`) computes a per-request signature from a user-defined template (supporting `{method}`, `{url}`, `{path}`, `{query}`, `{host}`, `{body}`, `{timestamp}`, `{nonce}` placeholders) using `crypto.createHmac`, encodes as hex or base64, and places the result in a configurable header or query parameter with optional prefix and companion timestamp/nonce headers.

All three types are supported end-to-end: new type definitions in `packages/core/src/types/auth.ts`, service implementations in `packages/shared/src/services/auth/`, wizard configuration sections in `AuthWizard.tsx`, and icon/label support in `AuthStoreGrid.tsx`. A new `OAuth2ServiceBase` shared base class consolidates token-endpoint mechanics (form-urlencoded POST, client auth method, response parsing, caching) across all three OAuth2 grant types. Pure PKCE helpers (`generatePkcePair`, `buildAuthorizationUrl`, `applyClientAuth`, `parseAuthorizationResponse`) in `packages/core/src/utils/oauth2.ts` use the Web Crypto API and are importable by both the wizard UI and the shared services.

Dead configuration was removed: `base64Encode` from `BaseAuth`, seven challenge-derived fields from `DigestAuth`, and unused `accessToken`/`tokenExpiresAt` from `OAuth2RefreshAuth`. The fabricated `|| 300` fallback for `expires_in` in `OAuth2RefreshService` was removed; `token_type` defaults to `'Bearer'` per RFC 6750 §6.1.1. All OAuth2 types require an explicit `clientAuthMethod` (`'basic'` = `client_secret_basic` / `'body'` = `client_secret_post`) with no preselected default. PKCE `state` is validated on exchange to prevent CSRF. Secrets are never logged.

## Request Processing

**Request Editor Spacing Polish & Params/Headers Copy-Paste Placement** *(updated: 2026-06-10 by implement)*
Source spec: [feature_request_editor_spacing.md](.spec-lite/features/feature_request_editor_spacing.md)
The Params and Headers tables no longer show a separate Copy/Paste toolbar row above the grid — the Copy and Paste icon buttons now live inside the table's **Actions** header cell, right-aligned next to the "Actions" label and sized to the header row. This reclaims a full row of vertical space above each table. Copy/Paste behavior is unchanged: Copy is disabled until at least one enabled, non-empty row exists, and Paste reads the clipboard and appends parsed key/value rows (JSON object, `key=value`, `key: value`, and url-encoded inputs all supported). Editor chrome spacing was also tightened to a consistent `px-2 py-2` scale across the method/URL toolbar and both the request and response tab-content areas in the VS Code and Web editors, with the dense response status bar kept at `py-1.5`.

**Save Keyboard Shortcut — Ctrl/Cmd+S in Request Editor** *(updated: 2026-06-11 by fix)*
Source spec: [feature_save_keyboard_shortcut.md](.spec-lite/features/feature_save_keyboard_shortcut.md)
`Ctrl+S` (Windows/Linux) or `Cmd+S` (macOS) saves the active request tab from anywhere inside the request editor — directly if the request is already linked to a collection, or by opening the Save wizard for unsaved requests. The shortcut is implemented as a container-scoped `onKeyDown` listener via the `useSaveShortcut` hook (exported from `@wave-client/core`), so it fires only when focus is inside the request editor and does not interfere with Environments, Flows, or other panes. The browser's native save dialog and VS Code's own Ctrl+S are suppressed via `preventDefault`, and the shortcut is automatically disabled while the Save wizard is open to prevent double-opening. Shortcut execution now mirrors Save-button eligibility: when save is disabled (for example, a clean collection-linked tab), Ctrl/Cmd+S is intercepted and silently ignored with no save banner. Params/Headers blur commits are now no-op when values are unchanged, so a successful keyboard save no longer re-activates the dirty indicator or triggers false "unsaved changes" prompts on the next click.

**Request Cancellation — Abort In-Flight Requests (FEAT-009)** *(updated: 2026-06-10 by implement)*
Source spec: [feature_request_cancellation.md](.spec-lite/features/feature_request_cancellation.md)
Users can abort an in-flight HTTP request from the editor instead of waiting for it to time out. While a request is processing, the **Send** button toggles to a **Cancel** (stop) button in both VS Code and Web; clicking it calls `http.cancelRequest(activeTabId)`. `HttpService` keeps an id-keyed `AbortController` registry: `execute` threads the controller's `signal` into the outbound axios call, and `cancel(requestId)` aborts and removes the matching controller (returning `false` for unknown/already-finished ids). The aborted request is mapped to a UI-friendly Cancelled response (`status: 0`, `statusText: 'Cancelled'`, empty body) rather than an error, and registry entries are always released in `finally`. Cancellation reaches the service over both platform channels — the VS Code `cancelHttpRequest` message (request/response correlated, replies with `{ cancelled }`) and the server route `POST /api/http/:id/cancel` — so the **server-side** outbound call is what actually aborts. The response viewer shows a distinct "Request cancelled" state, the request still appears in History (recorded at send time), and the editor is immediately ready for the next send. Cancellation events are logged with the request id and elapsed time (never bodies/headers).

**FEAT-FP-RPE-001: Request Processing Enhancements** *(updated: 2026-05-31 by fix)*
Source spec: [feature_request_processing_enhancements.md](.spec-lite/features/feature_request_processing_enhancements.md)
HTTP tabs now expose a request-side `Sent` tab that shows the exact on-wire request snapshot (final URL with query params, final headers, and a display-safe body) for the most recent send. Snapshot data is captured in `packages/shared/src/services/HttpService.ts` immediately before the request is dispatched and is returned on success, handled-internally auth responses, and error responses. The snapshot is stored ephemerally on `TabData.sentRequest` and is reset when a new send starts. The body is reduced to a single `{ text, format }` shape: form-data and url-encoded bodies are JSON-encoded (`format: 'json'`), binary/file bodies are summarised to metadata text (`format: 'text'`, never raw bytes), and raw bodies keep their text with a `format` hint (`json`/`xml`/`html`/`text`/`csv`) derived from the Content-Type. `RequestSent` in `packages/core/src/components/common/RequestSent.tsx` renders the method and full URL on a single line (method styled with the shared `getHttpMethodColor` palette), a headers section, and a body section that shows the format hint and the body text (no Content-Type row). Sections use distinct header/content backgrounds with light- and dark-mode support. The `Sent` tab is hidden until a request has actually been sent (no snapshot ⇒ no tab); both web and VS Code editors fall back to the first visible request tab if `Sent` was active when the snapshot was cleared.

Outgoing HTTP requests now get a default RFC 7231-style Wave Client `User-Agent` (`WAVE_CLIENT_USER_AGENT`) when no user-provided `user-agent` header exists (case-insensitive), while preserving custom user values unchanged.

Collections request operations now include Move and Duplicate actions on request rows. `RequestSaveWizard` supports `mode="move"` (title/description swap, request-name input hidden, path prefill), and selecting `Create New Collection` in move mode now clears any preselected collection and keeps the new-collection input editable (matching save mode behavior). Move now also accepts non-existing destination collection names: when the destination name is new, the move flow creates that collection and stores the request there before removing it from the source. `CollectionsPane` now handles move with same-path blocking, save-then-delete sequencing, cross-collection updates, and failure-safe notifications. Duplicate uses new `collectionParser` helpers to deep-copy request items with fresh IDs and collision-safe names (`Copy`, `Copy 2`, ...), then persists and updates the local store.

Coverage includes new and updated tests across core/shared/web, including: sent-request store behavior, `RequestSent` rendering, move-mode wizard behavior, move/duplicate collection flows, copy-name and deep-clone utilities, HttpService user-agent + sent-snapshot serialization, and web RequestEditor Sent-tab behavior.

**Request Editor Layout Compaction and Collapsible Side Panel Tabs** *(updated: 2026-06-08 by fix)*
The request editor in both web (`packages/web/src/components/RequestEditor.tsx`) and VS Code (`packages/vscode/src/webview/RequestEditor.tsx`) now uses tighter, consistent spacing across request and response toolbars, tab rows, metadata rows, and tab content containers so margins are less intrusive and visually balanced. The request section also has a left-side chevron control in the request tab row: users can collapse the entire request content area (Params/Headers/Body/Validation/Sent body) to reclaim vertical space for response/output inspection, then expand it back without losing the active request tab. The config side panel tabs (Collections/Flows/Test Lab/Wave Arena/History/Environments/Wave Store) now support toggle-close behavior in both platform wrappers (`packages/web/src/components/ConfigPanel.tsx`, `packages/vscode/src/webview/ConfigPanel.tsx`): clicking an already active tab closes its content panel, and the app shell collapses to the icon rail width (`64px`) while keeping the main editor/environment pane visible for more working real estate on smaller screens.

**History Real-Time Refresh — Auto-Reload After Persistence (FEAT-007)** *(updated: 2026-06-10 by implement)*
Source spec: [feature_history_realtime_refresh.md](.spec-lite/features/feature_history_realtime_refresh.md)
The History pane now updates immediately after each successful HTTP request send in both VS Code and Web clients. History persistence follows a pull-based sync helper (`useHistoryRefresh`): mutate persistence (`saveRequestToHistory`) and then reload from storage (`loadHistory` → `setHistory`) so the visible history list stays in sync with disk/server state without requiring reload. Manual retry still owns loading/error UI state; background refresh failures are logged and do not overwrite `historyLoadError`, so an already-working pane is not replaced with an error state. The clear/delete audit confirmed no additional stale flows: single-item delete already remains slice-consistent via `removeHistoryItem` after adapter-confirmed deletion.

**Response Download Fix — Consistent Download in VS Code & Web (FEAT-008)** *(updated: 2026-06-10 by implement)*
Source spec: [feature_response_download_fix.md](.spec-lite/features/feature_response_download_fix.md)
Response downloads now use a typed payload contract (`ResponseDownloadPayload`) end-to-end: `ResponseBody` emits `{ body, fileName, contentType }` with `body` always base64-normalized, and both platform request editors forward that payload type unchanged. In both apps, the download handler decodes payload base64 to bytes and calls `file.downloadResponse(bytes, fileName, contentType)`, so files are saved with the correct name and MIME type instead of hardcoded `response.json` and `"[object Object]"` content. On VS Code, `vsCodeAdapter.file.downloadResponse` now uses request/response correlation (`sendAndWait`) and sends the aligned wire shape `{ type: 'downloadResponse', data: { body, fileName, contentType } }`; `MessageHandler` reads that shape, writes decoded bytes through the save dialog, applies extension-aware filters, and replies with a correlated `downloadResponseResult` success/error message. On Web, the existing Blob/object-URL path in `webAdapter.downloadResponse` is used directly by the app handler.

**Content-Type Detection & Response Format Classification** *(updated: 2026-06-12 by fix)*
Content-type handling is now driven by a single canonical MIME registry in `packages/core/src/utils/common.ts` (`MIME_TYPE_REGISTRY`), from which both lookup directions are derived: file extension → MIME type (`getContentTypeFromFileName`, used for body file uploads) and MIME type (including legacy aliases like `text/xml`, `application/x-yaml`) → canonical download extension (`getExtensionFromContentType`). Coverage now spans modern text, code, data, image, audio/video, archive, font, certificate, and binary-serialization formats (e.g. AVIF/HEIC, WOFF2, WASM, NDJSON, Parquet, Zstandard); `.js` maps to `text/javascript` (RFC 9239) and `.yaml`/`.yml` to `application/yaml` (RFC 9512). Response format classification (`getResponseLanguage`) recognizes structured-syntax suffixes (`+json`, `+xml`, `+html`, `+csv`), treats all `text/*` and `message/*` plus known text-based `application/*` formats (JavaScript, YAML, TOML, GraphQL, NDJSON, JWT, form-urlencoded, etc.) as text, and **defaults unknown content types to binary** (download tile) instead of assuming text; a missing Content-Type header classifies as `none` (rendered as plain text, no content badge) so empty-body responses (e.g. 204) display normally. `getResponseContentType` extracts the Content-Type header case-insensitively and resolves multi-value headers to the first value. The request builder sets Content-Type via the shared `getContentTypeFromBodyMode` (the duplicate `getContentTypeForBody` in `requestBuilder.ts` was removed), raw-body language → Content-Type mapping is shared via `RAW_LANGUAGE_CONTENT_TYPES`, and the header-name autocomplete (`getCommonHeaderNames`) now includes modern security, fetch-metadata, WebSocket, proxy, and rate-limit headers. Generated download filenames no longer contain a double dot (`response_<timestamp>..json` → `response_<timestamp>.json`).

## WebSocket & SSE

**FEAT-010: Collection Persistence for WS/SSE** *(updated: 2026-04-15 by implement)*
WebSocket and SSE requests are now persisted in collection JSON files with full protocol fidelity. The save pipeline from `RequestEditor` through app handlers and adapters accepts `AnyCollectionRequest` (the discriminated union of HTTP, WS, and SSE) instead of HTTP-only `CollectionRequest`. `CollectionService` applies protocol-aware normalization on load (`normalizeRequestOnLoad`) and sanitization on save (`sanitizeRequestForSave`): WS payloads strip HTTP-only fields (`method`, `body`, `validation`); SSE payloads strip `validation` and default `method` to `GET`; legacy entries without a `protocol` field are normalized to HTTP. Runtime-only `sourceRef` metadata is never persisted. `extractRequestFromItem` and `requestToCollectionItem` in `collectionParser.ts` now use generic spreads to preserve all protocol-specific fields during round-trips. Mixed-protocol collections (HTTP + WS + SSE + legacy) load and save correctly with stable IDs. Covered by 15 new core tests (collectionParser round-trip) and 20 new shared tests (normalization, sanitization, persistence round-trip).

**FEAT-009: Web Adapter + Server Routes** *(updated: 2026-04-15 by implement)*
The web platform now has a full `IRealtimeAdapter` implementation via `createWebRealtimeAdapter` in `packages/web/src/adapters/webRealtimeAdapter.ts`, with synchronous handle creation (`connectWebSocket`, `connectSse`) and server-backed lifecycle operations (`/api/ws/connect`, `/api/ws/disconnect`, `/api/ws/send`, `/api/sse/connect`, `/api/sse/disconnect`). The server now exposes dedicated realtime route modules (`packages/server/src/routes/realtime-ws.ts`, `packages/server/src/routes/realtime-sse.ts`) that delegate connection management to shared services (`webSocketService`, `sseService`) and relay runtime updates through the existing push channel as `ws.*` and `sse.*` event types. `webAdapter.ts` now wires these push events into active handle registries (`wsHandles`, `sseHandles`) and exposes `adapter.realtime` so WS/SSE UI hooks work end-to-end in the web app. FEAT-009 includes comprehensive route tests (17 server tests) and web adapter tests (47 FEAT-009-focused tests across `webRealtimeAdapter.test.ts` and `webAdapter.test.ts`).

**FEAT-001: Protocol Types & Core Data Model** *(updated: 2026-04-14 by implement)*
Three request protocols are now defined as first-class types in Wave Client: `CollectionRequest` (HTTP, `protocol?: 'http'`), `WsCollectionRequest` (`protocol: 'ws'`), and `SseCollectionRequest` (`protocol: 'sse'`). These form the `AnyCollectionRequest` discriminated union used by `CollectionItem.request`. Type guard functions (`isHttpRequest`, `isWsRequest`, `isSseRequest`, `getRequestProtocol`) live in `packages/core/src/utils/requestTypeGuards.ts` and are exported from the core package index. All pre-existing collections without a `protocol` field are treated as HTTP (backward compatible — no migration needed). 35 unit tests in `packages/core/src/test/types/requestTypeGuards.test.ts` cover all guards, serialization round-trips, and backward compatibility.

**FEAT-002: Realtime State Slice** *(updated: 2026-04-14 by implement)*
Per-tab, in-memory realtime state for WebSocket and SSE connections is now part of the core Zustand store. New types in `packages/core/src/types/realtime.ts` define `ConnectionStatus`, `WsMessage`, `SseEvent`, `RealtimeTabState`, and `RealtimeStateByTabId`, all exported from `@wave-client/core`. A new Zustand slice (`createRealtimeSlice`) composed into `useAppStateStore` provides: idle state management (`ensureRealtimeTabState`, `getRealtimeState`, `setRealtimeStatus`, `setRealtimeConnectionId`, `setRealtimeResponseHeaders`, `setRealtimeError`, `resetRealtimeTabState`, `removeRealtimeTabState`); timeline actions (`appendWsMessage`, `appendSseEvent`, `clearRealtimeTimeline`, `setSseEventFilter`, `getFilteredSseEvents`); and automatic lifecycle wiring in `createRequestTabsSlice` so that loading, protocol-switching, clearing, and closing tabs always keep `RealtimeStateByTabId` consistent — WS/SSE tabs get a fresh idle entry on load, HTTP tabs never have one, protocol switches discard stale state. `TabData.protocol` is now typed as `RequestProtocol` (not a free-form string), with protocol-aware section defaults and request serialization (`getCollectionRequest()` returns the correct discriminated-union member). All realtime state is ephemeral (never persisted). 51 new unit tests cover TASK-001 through TASK-004.

**FEAT-005: SSE Service** *(updated: 2026-04-15 by implement)*
`SseService` in `packages/shared/src/services/SseService.ts` manages the full lifecycle of Server-Sent Events connections from the Node.js runtime (VS Code extension host or server). It opens SSE streams via `axios` with `responseType: 'stream'` (not the browser `EventSource` API — required for POST-based SSE, custom headers, and auth); parses the SSE wire protocol incrementally using an internal `SseFrameParser` class (handles `event:`, `data:`, `id:`, `retry:`, comments, partial chunks, `\r\n` endings, and all edge cases from the HTML Living Standard spec); routes parsed events and lifecycle transitions to event-driven `SseConnectionHandle` callbacks (`onEvent`, `onStatusChange`, `onError`, `onHeaders`); and closes connections via `disconnect()` using `AbortController`. Auth support (API key, Bearer, Basic — Digest is skipped with a warning), TLS/proxy via `storeService.getHttpsAgentForUrl()`, and default SSE headers (`Accept: text/event-stream`, `Cache-Control: no-cache`) are applied on every request. User-supplied headers override defaults. POST-based SSE with request body is supported. A maximum of 10 concurrent connections is enforced. Two new types were added to `@wave-client/core` and exported: `SseConnectionConfig` and `SseConnectionHandle`. The injectable `setSseAuthServiceFactory()` enables isolated unit testing. 44 unit tests in `packages/shared/src/test/services/SseService.test.ts` cover the frame parser (T01–T15), connection lifecycle (T16–T37), and auth/proxy integration (T38–T44).

**FEAT-004: WebSocket Service** *(updated: 2026-04-20 by fix)*
`WebSocketService` in `packages/shared/src/services/` manages the full lifecycle of WebSocket connections from the Node.js runtime (VS Code extension host or server). It opens connections with custom headers, query params, auth (API key, Bearer, Basic — Digest is skipped with a warning), and TLS/proxy settings via `storeService.getHttpsAgentForUrl()`; routes incoming messages and status transitions to event-driven `WsConnectionHandle` callbacks (`onMessage`, `onStatusChange`, `onError`, `onHeaders`); lets callers send text messages via `sendMessage()` (sent messages are emitted as `direction: 'sent'`); and closes connections via `disconnect()`. A maximum of 10 concurrent connections is enforced. Three new types were added to `@wave-client/core`: `Unsubscribe` (`() => void`), `WsConnectionConfig` (connection parameters), and `WsConnectionHandle` (event-driven handle). The injectable `setWsAuthServiceFactory()` enables isolated unit testing. A race condition fix now caches handshake response headers and immediately replays them to late `onHeaders` subscribers, so the Response Headers pane reliably populates even when the socket opens before UI listeners are attached. `useWsConnection.sendMessage()` now surfaces explicit user-facing errors for unavailable adapter/disconnected state, falls back to the tab's stored `connectionId` when the local hook handle is unavailable (for example after component remount), and appends a local sent message on successful send so the Messages pane updates immediately. To avoid duplicate sent entries, WS push relays now ignore `direction: 'sent'` echoes in both server and VS Code bridge paths. 28 unit tests in `packages/shared/src/test/services/WebSocketService.test.ts`, 19 hook tests in `packages/core/src/test/hooks/useWsConnection.test.ts`, 11 server route tests in `packages/server/src/test/routes/realtime-ws.test.ts`, and 24 VS Code handler tests in `packages/vscode/src/test/handlers/MessageHandlerRealtime.test.ts` cover public methods, auth resolution, connection limits, race handling, and send-path robustness.

**FEAT-008: VS Code Adapter — Realtime Integration** *(updated: 2026-04-15 by implement)*
The VS Code platform adapter now fully implements `IRealtimeAdapter` so the WS/SSE UI (FEAT-006, FEAT-007) works end-to-end inside VS Code. **FEAT-003: Protocol Selector & Conditional Editor Shell** *(updated: 2026-05-21 by fix)*

## Validation

**FEAT-001: Validation Engine Enhancement** *(updated: 2026-05-26 by fix)*
The validation engine in `packages/shared/src/utils/validationEngine.ts` now uses `jsonpath-plus` for full JSONPath expression evaluation (recursive descent `$..`, wildcards `[*]`, array slicing, filter expressions) and `ajv` for real JSON Schema validation (draft-07 compatible). The old naive dot-notation splitter and parse-only stub have been replaced. A new `validateJsonSchemaString(schemaString)` helper is exported from `@wave-client/shared` — it returns `{ valid: boolean; errors?: string[] }` and is intended for UI-side pre-validation of user-provided schema strings before saving a rule. Regression coverage was expanded in `validationEngine.test.ts` to cover these JSON-object operator paths and fallback behavior.

Source spec: [feature_http_file_transformer.md](features/feature_http_file_transformer.md)
